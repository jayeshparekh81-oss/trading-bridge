## 2 — Proposed Architecture

### 2.1 Decision principle

> **Reuse `DhanHistoricalClient` (the per-user async path) as the single fetcher; add a persistence layer + scheduled backfill; have backtest's `_fetch_dhan_candles` migrate to it via a thin bridge.** Do NOT build a 4th Dhan client. Do NOT touch the sacred `DhanBroker`.

### 2.2 New module layout (all NEW files — adheres to new-files-only rule)

```
backend/app/services/historical_candles/
    __init__.py
    repository.py            # DB read/write for the new candles table
    backfill_service.py      # Orchestrates fetch + store + idempotency
    fetch_orchestrator.py    # Wraps DhanHistoricalClient, adds chunking + quality gate
    symbol_bridge.py         # Reconciles KNOWN_SYMBOLS ↔ DhanBroker scrip master
    schema_bridge.py         # schemas.candle.Candle ↔ strategy_engine.schema.ohlcv.Candle

    historical_candle.py     # New SQLAlchemy model (NEW FILE — does not touch existing models)

backend/migrations/versions/
    <timestamp>_add_historical_candles.py  # NEW — designed in §4, not created yet

backend/app/tasks/
    historical_backfill_tasks.py  # NEW Celery task module — not registered to beat until Phase 3

backend/scripts/

```

**Why a new `services/historical_candles/` namespace** instead of extending `strategy_engine/data_provider/`?

- `data_provider/` is owned by the strategy_engine sub-package. The new layer crosses chart, indicators, AND backtest — pulling it into `services/` matches the layering of `services/chart_redis.py`, `services/indicator_candles.py`.
- Keeps the `data_provider/` package's file cache + KNOWN_SYMBOLS intact as a fallback the new orchestrator can prefer when the DB is cold.
- Phase 4 deprecates `data_provider/` cleanly after migration.

### 2.3 Data fetcher service shape

```python
# fetch_orchestrator.py — pseudocode, NOT real code yet
class HistoricalFetchOrchestrator:
    async def fetch(
        self,
        *,
        user: User,
        db: AsyncSession,
        symbol: str,
        exchange: Exchange,
        timeframe: Timeframe,
        from_ts: datetime,
        to_ts: datetime,
        prefer_storage: bool = True,
    ) -> tuple[list[Candle], FetchSource]:
        # 1. If prefer_storage: check DB candles table for window coverage.
        #    Coverage = contiguous closed bars between from_ts and to_ts.
        #    Return early if 100% covered.
        # 2. Compute gaps (sub-windows not in DB).
        # 3. For each gap: chunk into ≤90d intraday / ≤5y daily slices.
        # 4. For each chunk:
        #    a. resolve_dhan_credentials(user)
        #    b. symbol_bridge.resolve(symbol, exchange) — try scrip-master
        #       first (DhanBroker), fall back to KNOWN_SYMBOLS.
        #    c. DhanHistoricalClient(...).get_historical_ohlc(...)
        #    d. validate_candles() — Phase 11 quality gate.
        #    e. repository.upsert_candles(...) — idempotent by
        #       (symbol, exchange, timeframe, timestamp) PK.
        # 5. Re-read the full window from DB → return.
```

**Source enum** so callers/log lines can attribute provenance:

```python
class FetchSource(StrEnum):
    DB_HIT = "db_hit"                # 100% from DB
    DB_PARTIAL = "db_partial"        # some DB + some live
    LIVE_DHAN = "live_dhan"          # 0% DB, all live (and now persisted)
    KNOWN_SYMBOLS_FALLBACK = "known_symbols_fallback"  # scrip master miss
```

### 2.4 Storage schema (proposed — see §4 for full migration outline)

**Table: `historical_candles`**

| Column | Type | Notes |
|---|---|---|
| `symbol` | `text` | Canonical Dhan trading symbol (upper, alias-resolved). |
| `exchange` | `text` | NSE / BSE / NFO / BFO / MCX / CDS. Matches `Exchange` enum values. |
| `timeframe` | `text` | `1m / 5m / 15m / 1h / 1d`. Matches `Timeframe` enum. |
| `timestamp` | `timestamptz` | Bar-OPEN time, UTC. |
| `open` | `numeric(18,4)` | Decimal-friendly. Loss-free vs the `Decimal` in `schemas.candle`. |
| `high` | `numeric(18,4)` | |
| `low` | `numeric(18,4)` | |
| `close` | `numeric(18,4)` | |
| `volume` | `bigint` | NOT NULL, default 0. |
| `dhan_security_id` | `text` | Persisted alongside row to detect Dhan id rotation later. |
| `source` | `text` | `dhan_v2_historical` / `dhan_v2_intraday`. Audit trail. |
| `fetched_at` | `timestamptz` | When we wrote it. |
| `fetched_by_user_id` | `uuid` | FK to `users.id` of the first user who triggered the pull. Allows attribution + later compliance. |
| `quality_score` | `numeric(5,2)` | Phase 11 score at write time. Lets future readers re-filter. |

**Primary key:** `(symbol, exchange, timeframe, timestamp)` — natural composite, lets `INSERT ... ON CONFLICT DO NOTHING` give us idempotent upserts without uuid bloat.

**Indexes:**
- `idx_hc_lookup` on `(symbol, exchange, timeframe, timestamp DESC)` — covers the window query.
- `idx_hc_freshness` on `(timeframe, fetched_at)` partial WHERE `timeframe IN ('1m','5m','15m')` — for "stale intraday" sweep.

**Partitioning (deferred to Phase 5):** monthly partitions on `timestamp` for `1m` and `5m`. Skipped in v1 — at 507 symbols × 1m × 252 sessions × 375 bars ≈ 47M rows/year; single partition is fine to start, partition later when query times degrade. Founder gate item.

### 2.5 Rate limit handling

- **Reuse** `DhanHistoricalClient`'s per-user `rate:dhan_historical:{user_id}` Redis key. No change.
- **New** for backfill: dedicated **system user** (env-provided creds, NOT a real user's) for the scheduled backfill — its own rate-limit key keeps user-driven fetches isolated.
- Backfill script paces itself client-side: `asyncio.Semaphore(4)` (leave 1 req/s headroom) + jittered sleep between symbols.

### 2.6 Idempotency

Two layers:

1. **Row-level:** PK on `(symbol, exchange, timeframe, timestamp)` + `INSERT ... ON CONFLICT DO NOTHING`. Duplicate fetches no-op.
2. **Job-level:** `historical_backfill_jobs` table (designed in §4) — `(symbol, exchange, timeframe, window_start, window_end)` uniqueness. Worker skips a job that's already `status='succeeded'` within the freshness TTL.

### 2.7 JWT rotation mid-fetch

The chunked path is vulnerable: a 5-year daily backfill might span an access-token refresh.

**Plan:**
- Re-resolve credentials at **the start of each chunk**, not once per job. `DhanHistoricalClient` is per-request anyway — extending to per-chunk is one extra DB read.
- On `BrokerAuthError` mid-fetch: pause the worker (no retry loop with stale token), publish a `BROKER_DISCONNECTED` event (same contract as chart), surface a Hinglish "Dhan session expire ho gayi — relink karo" admin email if this is the system-user backfill.

### 2.8 Error + retry policy

| Error from `DhanHistoricalClient` | Per-chunk action | Per-job action |
|---|---|---|
| `BrokerAuthError` | Stop chunk loop. | Mark job `failed_auth`, do NOT retry. Notify. |
| `BrokerRateLimitError` (local) | Sleep `_LOCAL_RATE_RETRY_SLEEP_S * 2`, retry chunk once. | Job survives. |
| `BrokerRateLimitError` (upstream 429) | Already retried inside client. | Job retries with backoff `[1m, 5m, 15m]`. |
| `BrokerUpstreamError` (5xx / non-JSON) | Re-raise. | Retry up to 3× over 1h with jitter. After 3: `failed_upstream`. |
| `BrokerInvalidParamsError` | Re-raise. | `failed_params` — do NOT retry. Symbol likely delisted. |
| Network timeout | Already retried inside client. | Job retries identical to upstream. |

### 2.9 Backfill strategy

**Initial pull (one-shot per symbol):**

- Daily timeframe: 5y window in 1 Dhan call → small payload, ~200 rows/symbol → 507 symbols ≈ 102k daily rows. <30 min wall-clock at 5 req/s.
- 1h intraday: 90d × 6 calls covers ~1.5 years → ~6h budget for 507 symbols at 4 req/s with jitter.
- 5m intraday: 90d window per call, walk back 2 years → ~8 calls/symbol × 507 = 4k calls ≈ 14 min at 5 req/s but practical wall-clock with quality gate retries ~1h.
- 1m intraday: defer (massive storage; only fetch on demand for the 5 active templates).

**Ongoing updates:**

- Celery beat schedule (NOT REGISTERED YET — Phase 3):
  - `1d`: every weekday 17:00 IST — fetch yesterday→today for all symbols. ~507 calls.
  - `5m / 15m / 1h`: every weekday 15:45 IST (post-close) — fetch today's window.
  - `1m`: on-demand only (template-triggered).

## 3 — Integration Points

### 3.1 Backtest engine consumption

**Today** (`strategy_engine/api/backtest.py:508-553`):
```
candles_request → fetch_historical_candles (PROCESS TOKEN) → quality gate → engine
```

**Phase 4 wiring:**
```
candles_request → HistoricalFetchOrchestrator.fetch(user, ...) → schema_bridge → engine
```

The `BacktestRunRequest.candles_request` schema **stays identical** — the change is internal. The `candles_source` literal gains a third value (`"dhan_historical_persisted"`) only if Jayesh wants the UI to differentiate; otherwise reuse `"dhan_historical"`.

### 3.2 Fallback policy

| Situation | Decision |
|---|---|
| Symbol fully in DB | Serve from DB, no Dhan call. |
| Symbol partially in DB (recent gap) | Fetch only the gap, splice. |
| Symbol absent + scrip-master hit | Live fetch, persist, serve. |
| Symbol absent + scrip-master miss + `KNOWN_SYMBOLS` hit | Use bundled metadata, fetch, persist, serve. |
| Symbol absent + both miss | Raise `BrokerInvalidParamsError` (404 to UI). Do NOT silently fall back to synthetic — backtest realism requires honesty. |
| Dhan auth fail | 401 Hinglish; do NOT silently fall back to synthetic. |
| Dhan rate-limit exhausted | 429 Hinglish with retry-after. |
| Quality score <40 | 422 (same as today). |

**No silent synthetic fallback** when `candles_request` is set — that's a footgun. Synthetic stays the explicit default when `candles_request is None`.

### 3.3 Indicator service consumption (Phase 5+)

Once the DB layer is stable, `services/indicator_candles.py:fetch_closed_candles` can switch its inner `client_factory` call to the orchestrator. Same per-user creds, same closed-candle filter. Cuts indicator latency dramatically.

### 3.4 Chart endpoint consumption (Phase 5+)

Chart history can optionally consult the DB before hitting Dhan, reducing chart load on Dhan. The Redis 5-min cache stays in front of both DB and Dhan.

## 5 — Risk Analysis

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Dhan rate-limit exhausted under backfill load → user-facing 429s | Medium | High | System-user creds with its own rate-key, backfill Semaphore(4), no system-user backfill during market hours 09:00–15:30 IST. |
| Partial fetch leaves DB with sparse coverage → backtests run on holes | Medium | Medium | Job-level idempotency table tracks `window_start`/`window_end`; orchestrator computes gap before serving. |
| Schema bridge bug loses precision (Decimal → float → Decimal) | Low | Medium | Bridge tests pin against canonical fixtures; float→Decimal goes via `str()` (already the pattern in `dhan_historical._money`). |
| Storage growth runaway on 1m candles | Low (deferred) | Medium | Phase 5 partitioning. 1m on-demand only in v1. |
| Conflicts with existing Celery beat schedule | Low | Medium | New `historical_backfill_tasks.py` module; new beat entries added under a feature flag (`enable_historical_backfill_beat`). Off by default. |
| Drift between bundled `KNOWN_SYMBOLS` and Dhan scrip master | Medium | Low | Backfill prefers live scrip master; `KNOWN_SYMBOLS` is fallback only. |
| Quality-gate false positives reject otherwise usable bars | Medium | Low | Quality score stored on row; readers can re-filter; threshold tunable via Settings. |
| Multi-tenant data leak (one user's fetch visible to another) | Low | High | `historical_candles` is **shared infrastructure data** — same OHLC for everyone. `fetched_by_user_id` is for attribution, not access control. Document this in row docstring. |
| Impact on live BSE LTD strategy 89423ecc (real-money) | Very low | Catastrophic | This work touches ZERO sacred files. Sacred zone has been re-verified above. No imports from `strategy_executor`, `direct_exit`, `webhook`, `kill_switch`. New code runs on isolated Celery queue. |
| Mid-fetch JWT rotation losing chunks | Low | Low | Re-resolve creds per chunk; auth fail aborts job cleanly (no half-written window state). |
| Two Candle classes proliferate further | Medium | Low | Bridge functions only — no inheritance. Phase 6 unification flagged in PATCH_INSTRUCTIONS. |
| `/tmp` file cache conflicts with new DB cache | Low | Low | Phase 4 deletes `data_provider/cache.py` after backtest cuts over. Until then both can coexist (different paths). |
| Backfill triggered before alembic migration applies → 500 on first request | Medium | Medium | Two deploy hard-stops per CLAUDE.md: founder gate before migration, founder gate before container restart. |

## 7 — Open Decisions for Founder

Inline so you can mark them up directly. **A / B / C choices, with my recommendation tagged.**

### Q1 — Storage cost vs API cost tradeoff

Background: full backfill of 507 symbols across 1d+1h+5m ≈ 20M rows ≈ ~3.5 GB on the existing EC2 Postgres.

- **A (Recommended)** — Backfill 1d + 1h + 5m for the full 507-symbol `KNOWN_SYMBOLS` set; defer 1m to on-demand. Storage hit ~3.5 GB; eliminates Dhan calls for the vast majority of backtest runs.
- B — Backfill only the 20 popular symbols (Phase 3 list); rest stay on-demand. Storage <500 MB; but cold-start latency on tail symbols.
- C — Skip backfill entirely; cache-on-read only. Lowest storage; highest Dhan-call rate; user-facing latency on first backtest of any unseen symbol.

### Q2 — Which symbols to backfill first

- **A (Recommended)** — Phase 3 list above (20 symbols, top liquidity). Founder edits the list before Phase 3 starts.
- B — All 7 indices + top 50 F&O stocks.
- C — Full `KNOWN_SYMBOLS` set in one go (507 symbols, ~6h wall-clock).

### Q3 — On-demand fetch vs scheduled backfill (hot path)

For symbols NOT in nightly backfill:

- **A (Recommended)** — User-driven backtest triggers a synchronous fetch (with quality gate); future requests hit DB. Latency: ~3s first time, ~50ms cached.
- B — User-driven backtest returns 503 "data not available, scheduled for tomorrow's backfill"; queues the symbol for next nightly run.
- C — Hybrid: synchronous for ≤90d windows, queue for larger.

### Q4 — Hot-path execution context: live Celery worker vs batch script

For the scheduled backfill:

- **A (Recommended)** — Dedicated Celery beat entry → Celery worker on existing infra; isolated queue name `historical_backfill` so it never blocks orders. Off by default; flag `enable_historical_backfill_beat=False` until founder flips.
- B — Cron-triggered standalone Python script (Docker oneshot). Simpler isolation; no Celery dependency.
- C — Manual ops invocation only (script in `backend/scripts/`); no automation.

### Q5 — Per-user creds vs system creds for nightly backfill

- **A (Recommended)** — System creds (founder's own Dhan account, env-provided as `BACKFILL_DHAN_*`). Per-user rate-limit on user-driven path stays untouched. Required so backfill doesn't burn paying users' 100k/day quota.
- B — Rotate through users who have linked Dhan (round-robin). Distributes Dhan quota use but multi-tenant complexity high.
- C — Per-user backfill only when the user submits a backtest (no system backfill).

### Q6 — Quality gate threshold for backfill vs backtest

Today: backtest rejects <40. Should backfill also gate, or store + tag with quality_score and let readers decide?

- **A (Recommended)** — Backfill writes everything with `quality_score` column populated; readers (backtest) apply the gate. Lets a future relaxed quality threshold (e.g., "≥20 for daily, ≥40 for intraday") not require re-fetch.
- B — Backfill mirrors backtest's <40 gate — never persist low-quality data.
- C — Backfill writes everything; remove the quality gate from backtest too.

### Q7 — Migration sequencing relative to BSE LTD strategy 89423ecc

The new table + indexes will lock the schema briefly during ALTER. Postgres on EC2 single-node.

- **A (Recommended)** — Deploy migration during pre-market window (~07:00 IST). New table only (no ALTER on existing tables) → instant. Zero blast radius for sacred live strategy.
- B — Deploy after market close (~16:00 IST) per usual change window.
- C — Blue-green: spin up replica, migrate replica, cut over.

### Q8 — Data retention policy

- **A (Recommended)** — Retain forever (it's append-only historical). Storage growth is bounded by symbol-set growth, not user count.
- B — 2-year rolling window; older data deleted nightly.
- C — Tiered: 5y daily, 2y hourly, 6mo 5m.
