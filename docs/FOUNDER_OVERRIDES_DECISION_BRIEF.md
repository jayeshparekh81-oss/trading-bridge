## §0 — Cross-cutting context up-front

Three things changed how I'd rate these options today vs. when Queue BB
drafted them:

1. **The engine ALREADY computes & stores sub-outputs.**
   `backend/app/strategy_engine/backtest/indicator_runner.py:86-93` writes
   dotted keys `series_by_id["macd_12_26_9.signal"]` etc. for every
   multi-output indicator. The blocker is purely the **schema's
   lower-snake-case validator on `IndicatorConfig.id`** (no dots) PLUS the
   `StrategyJSON` referential-integrity check (every `IndicatorCondition.
   left/right` must equal some `IndicatorConfig.id`). So options that work
   *with* the existing dotted-key engine plumbing are dramatically cheaper
   than they looked.
2. **`CandlePattern` enum already supports DOJI, HAMMER, ENGULFING,
   SHOOTING_STAR, BULLISH, BEARISH** (`strategy.py:156-164`). Phase 2
   work is grammar-only — no schema change for those three patterns.
   `HANGING_MAN` is NOT enumerated.
3. **No `bars_back` primitive in schema** — `IndicatorCondition` has no
   lookback offset field. Any multi-bar lookback path requires a schema
   addition.

## §1 — Decision A: Sub-output references

### A. Context
Templates reference bare sub-output names in prose (`macd_line`,
`signal_line`, `macd_histogram`, `bb_upper`, `bb_middle`, `bb_lower`).
The schema only validates that `IndicatorCondition.left/right` is in
`indicators[*].id`. Currently the translator emits these references but
doesn't declare matching IndicatorConfig entries → `ValidationError:
Conditions reference indicator ids not declared in 'indicators'`.

**Templates blocked (4):** `macd-trend-signal`, `rsi-macd-confluence`,
`bb-rsi-oversold`, `orb-15min` (`orb_15_high` is the ORB equivalent of
sub-output).

### B. Options analysis

| Option | What it changes | Files touched | Risk to live | Time | Maintenance |
|---|---|---|---|---:|---|
| **A1. Dotted-id schema support** | Relax `IndicatorConfig.id` lower-snake validator to allow dots; update referential integrity to walk dotted ids; translator emits `macd_12_26_9.signal` | `strategy.py` (validator), `parser.py`/`field_mappers.py` (translator) | **MED** — schema change ripples to any consumer that re-validates strategies (live order routing re-validates on each request) | 2–3 h | LOW — engine already keys this way, future multi-output indicators "just work" |
| **A2. Alias-key extension** (Queue GG option β) | Post-compute step in `precompute_indicators` aliases dotted keys to underscore-keyed twins (`.signal` → `_signal_`); translator auto-declares `IndicatorConfig(id="macd_signal_12_26_9", type="macd", params=...)` so schema passes; aliases make engine lookup succeed | `indicator_runner.py` (8 lines), `parser.py`/`field_mappers.py` (translator) | **LOW–MED** — `indicator_runner` is on the live trading path but the change is additive-only (extra dict keys, no removal) | 1–1.5 h | MED — every multi-output indicator added later needs an alias rule |
| **A3. Per-template overrides** | Hand-write 4 `StrategyJSON` blobs in `override_registry` for `macd-trend-signal`, `rsi-macd-confluence`, `bb-rsi-oversold`, `orb-15min`. Zero translator/engine change | `override_registry.py` only (`+ ~30 lines/template`) | **LOWEST** — registry-only, never touched by live | 30–45 min | HIGH — every new sub-output template needs its own override forever |

### C. Cross-cutting impacts
- **A1 enables Decision D (colour-flip prose)** to also use dotted refs if
  Supertrend etc. ever exposes a "direction" sub-output. Otherwise standalone.
- **A1 enables CandleCondition's `pattern` to be addressed by id later** if
  CandleCondition gains an `id` field for "fired this bar" semantics. Speculative.
- **A2's alias step would coexist cleanly with later A1 adoption** — aliases
  become redundant but harmless.
- **A3 has zero cross-cutting impact** (the point of the option).

### D. Recommendation
**A2 (alias-key extension)** as the next step, with **A1 (dotted-id schema)
as the planned follow-up** once the live-trading path is verified stable
on the alias approach. A3 is appropriate ONLY if we're not extending the
translator at all in the next 2 weeks — its maintenance burden compounds
as templates are added.

**Reason:** A2 unlocks the same 4 templates as A1 with materially less
schema-validation blast radius. Schema changes (A1) touch the
`re-validate-on-every-live-request` path; alias changes (A2) only touch
the precompute pass that runs once per backtest. Defer A1 until live
metrics confirm A2 didn't regress anything.

## §2 — Decision B: Candle-pattern grammar

### A. Context
Three templates use candle-pattern prose (`engulfing-candle-reversal`,
`doji-reversal`, `hammer-hanging-man-pattern`). The schema's
`CandleCondition` and `CandlePattern` enum (DOJI, HAMMER, ENGULFING,
SHOOTING_STAR, BULLISH, BEARISH) already model 5 of the 6 patterns the
templates need. Parser doesn't have grammar to recognise these in prose.

**Templates blocked (3):** `engulfing-candle-reversal`, `doji-reversal`,
`hammer-hanging-man-pattern`.

### B. Options analysis

| Option | What it changes | Files touched | Risk | Time | Maintenance |
|---|---|---|---|---:|---|
| **B1. Translator grammar extension only** | Add regex patterns: "bullish engulfing pattern" → `CandleCondition(pattern=ENGULFING)`, "doji" → DOJI, "hammer" → HAMMER. `hanging-man` skipped (no enum) or mapped to HAMMER with a flag. | `field_mappers.py` (translator) | **LOW** — translator-only, no live-path code | 1–1.5 h | LOW — pattern names are stable PSPL vocabulary |
| **B2. Schema primitive expansion + grammar** | Add `BULLISH_ENGULFING`, `BEARISH_ENGULFING`, `HANGING_MAN` to `CandlePattern` enum + engine detection logic for the directional variants; then grammar | `strategy.py` (enum), `indicator_runner.py` or `pattern_detector.py` (engine), `field_mappers.py` (translator) | **MED** — engine detection is on the live trading path | 3–4 h | LOW |
| **B3. Per-template overrides** | Hand-write 3 `StrategyJSON` blobs | `override_registry.py` (`+ ~40 lines/template` — engulfing detection logic baked in) | **LOWEST** | 1–1.5 h | HIGH — every candle-pattern template needs its own override |

### C. Cross-cutting impacts
- **B1 unblocks 3 templates immediately;** the directional distinction
  (bullish vs bearish engulfing) is lost — the engine matches "any
  engulfing" until B2 ships.
- **B2 is the proper fix** but the engine detection work (multi-bar
  pattern matching with previous-bar context) is non-trivial.
- **B3 has no cross-cutting impact.**

### D. Recommendation
**B1 (translator grammar only)** as the next step. The directional
ambiguity for engulfing is acceptable because all 3 templates use the
pattern as a *signal* combined with other conditions (RSI threshold,
trend filter) that constrain direction.

**Reason:** B1 unlocks 3 templates in 1.5 h with no live-trading risk.
B2's engine work is properly scoped as a future enhancement once the
templates are observed in use. B3 doesn't scale.

## §3 — Decision C: Divergence detection

### A. Context
Three templates use bullish/bearish divergence prose (`rsi-divergence`,
`macd-divergence`, `obv-divergence`). The schema has NO primitive for
"price prints lower low AND indicator prints higher low" — neither
`IndicatorCondition` nor `CandleCondition` model the multi-bar local
extrema pattern divergence requires.

**Templates blocked (3):** `rsi-divergence`, `macd-divergence`, `obv-divergence`.

### B. Options analysis

| Option | What it changes | Files touched | Risk | Time | Maintenance |
|---|---|---|---|---:|---|
| **C1. New `DivergenceCondition` schema primitive + engine detection** | Add `DivergenceCondition(type="divergence", indicator, price_extremum, indicator_extremum, lookback_bars)`; engine implements local-extrema detection over `lookback_bars` window; translator grammar parses "price prints lower low... indicator prints higher low" | `strategy.py` (new condition type, discriminated union expansion), engine detection module (new file), `field_mappers.py` (grammar) | **MED–HIGH** — touches the discriminated union which every condition evaluator branches on | 6–10 h | LOW once shipped |
| **C2. Indicator-side detection** (new `_divergence` indicator) | Engine ships a new indicator `rsi_divergence_14_25` whose value is 1 when bullish divergence is detected, 0 otherwise. Translator parses divergence prose to a comparison against this new indicator | new file `indicators/calculations/_divergence.py` (already exists — see registry.py line 440 referencing `rsi_divergence` / `macd_divergence` / `obv_divergence`), registry entries, translator | **LOW** — additive indicators, no schema change | 3–4 h | MED — params encode the divergence rules |
| **C3. Per-template overrides** | Each override hand-builds the divergence comparison using existing primitives (won't actually detect divergence — would have to skip the divergence clause or stub it). | `override_registry.py` | **LOWEST** but **semantically wrong** — templates lose their core entry signal | 1 h | N/A — strategies don't behave as advertised |
| **C4. Deactivate in seed** | Set `is_active=false` for these 3 templates in `strategy_templates_seed.json` (NB: this is FORBIDDEN under Queue GG guardrails — needs founder OK) | seed file | **LOW** | 15 min | N/A — re-activate when divergence ships |

### C. Cross-cutting impacts
- **C2's `_divergence` indicator path may already be partially built** —
  `registry.py` line 440 references `rsi_divergence` / `macd_divergence` /
  `obv_divergence` in a comment. Investigation needed.
- **C1's discriminated-union expansion is shared infrastructure** — once
  done, future multi-bar primitives (multi-bar lookback per Queue GG
  Phase 3, candle-pattern sequences) get easier.
- **C4 contradicts the "no seed edits" Queue GG/HH guardrail** but is
  on the table for a future founder-supervised change.

### D. Recommendation
**C2 (indicator-side detection)** because the divergence calculation
modules appear to already be drafted in `_divergence.py`. Hand-build the
3 registry entries + translator grammar to recognise "X prints lower
low... Y prints higher low" → comparison against the new indicator.

**Reason:** C2 ships divergence as a real signal (not a stub), without
the schema discriminated-union surgery C1 demands. C4 (deactivate) is the
honest fallback if `_divergence.py` turns out to be a skeleton — flag
back to founder.

## §4 — Decision D: Colour-flip prose (Supertrend / Hull-MA / Triple-EMA)

### A. Context
Templates reference UI-visual semantics ("supertrend flips to bullish",
"hull_ma_21 colour flips from red to green") that map to numeric
primitives the schema *does* support, but the seed prose is rendered for
human readability — not parsing.

**Templates blocked (3):** `supertrend-rider`, `hull-ma-trend`,
`triple-ema-crossover` (also has a "in the last 2 bars" lookback,
overlap with Decision E / Queue GG Phase 3).

### B. Options analysis

| Option | What it changes | Files touched | Risk | Time | Maintenance |
|---|---|---|---|---:|---|
| **D1. Translator grammar interprets the prose** | Add patterns: "supertrend flips to bullish" → `IndicatorCondition(left=close, op=CROSSOVER, right=supertrend_10_3)`; "hull_ma_21 colour flips from red to green" → `IndicatorCondition(left=hull_ma_21, op=CROSSOVER, right=hull_ma_21_lagged_1)` (would need a 1-bar lag indicator — adds scope) | `field_mappers.py` (translator) + maybe new lagged-series primitive | **LOW** (translator-only) | 2–3 h for the simpler patterns; longer if lag indicator needed | MED — visual prose is open-ended |
| **D2. Prose rewrite in seed** | Edit the seed templates to use numeric primitives directly: `"supertrend flips to bullish"` → `"close crosses above supertrend_10_3"`. The translator already handles the rewritten form. **FORBIDDEN under Queue GG/HH guardrails** — needs founder OK to edit seed | seed file (3 templates) | **LOW** | 30 min | LOW |
| **D3. Per-template overrides** | Hand-write 3 `StrategyJSON` blobs that bake in the intended numeric semantics | `override_registry.py` | **LOWEST** | 1–1.5 h | HIGH |

### C. Cross-cutting impacts
- **D2 is the philosophically correct fix** — the seed prose was authored
  for users to read in the strategy-card UI; if the parser can't
  understand it, the prose is also misleading to the user about what the
  strategy actually does. But the no-seed-edits guardrail blocks it.
- **D1 risks growing the grammar surface every time someone writes new
  visual prose.** Bounded if we commit to a finite vocabulary.
- **D3 has no cross-cutting impact.**

### D. Recommendation
**D2 (seed rewrite)** is the right answer but requires a founder-approved
exception to the no-seed-edits guardrail. If that's a no-go, fall back to
**D3 (overrides)** for these 3 templates — they're outliers and the
visual-prose grammar (D1) is hard to bound.

**Reason:** Visual semantics ("flips green") are UI metaphors, not formal
operators. Either teach the schema to understand them (D1 — complex and
unbounded) or rewrite the seed to use formal operators (D2 — touches a
deliberately-frozen file). Overrides are the realistic stopgap until D2
is OKed.
