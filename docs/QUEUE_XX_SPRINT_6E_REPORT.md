## 1. Dual scoreboard methodology

Each verified indicator gets two tier classifications:

- **`tier_pine`** — TRADETRI's classification against the Pine docs /
  hand-rolled Pine convention. This is the **authoritative tier** for
  customer-facing decisions (TradingView UI parity).
- **`tier_talib`** — TRADETRI's classification against TA-Lib's
  industry-default reference. Shown for customers who cross-check via
  TA-Lib downstream Python libraries.

For TA-Lib-missing indicators (Donchian, Ichimoku, all custom Pack-2-16
+ session-aware), `tier_talib = (no talib)` — there's no industry-library
reference to compare against.

## 10. Sprint 6e framework lesson (lesson #15 for the chain)

**Maintaining two scoreboards (vs Pine, vs talib) is the only honest answer**
to "is TRADETRI correct?" Both classifications are useful for different
audiences:

- **vs Pine** — answers "do TRADETRI values match what a TradingView user
  sees?" Yes (78 A, 14 B, 4 D out of 96).
- **vs talib** — answers "do TRADETRI values match what a Python TA-Lib
  user would compute?" Mostly yes, but 6 indicators (Aroon family + chande
  + chaikin) follow Pine docs over TA-Lib's internal convention.

Future Sprint 7+ framework v3 should ship both classifications by default
rather than forcing a single-reference choice.
